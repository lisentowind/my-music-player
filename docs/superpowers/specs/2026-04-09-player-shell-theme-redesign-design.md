# 播放器壳层与主题系统重设计

Date: 2026-04-09

## 摘要

在现有 Tauri + Vue 3 音乐播放器基础上，完成一次聚焦“布局、视觉、播放内核、主题能力”的重设计。目标是将当前偏散乱、信息层级不清的液态玻璃界面，升级为更成熟的桌面播放器产品壳：左侧固定、顶部固定、右侧内容区独立滚动，底部 Dock 更紧凑；播放器底层改用成熟第三方开源库；全局支持亮色、暗色、跟随系统、预设主题色以及自由取色。

## 目标

- 左侧导航固定，顶部栏固定，右侧内容区独立滚动
- 重新设计整体 UI，提升层级、留白、可读性和桌面应用完成度
- 图标统一改为 Iconify，不再用文字替代操作语义
- 播放底层从原生 `HTMLAudioElement` 包装迁移到 `Howler.js`
- 新增主题系统：
  - 亮色
  - 暗色
  - 跟随系统
  - 预设主题色
  - 自由取色
- 封装更多可复用的 UI 基础组件，减少样式和交互重复

## 非目标

- 不接入在线音乐服务或后端
- 不实现歌词、均衡器、波形可视化
- 不引入大型 UI 框架
- 不把本轮扩展成完整设置中心或复杂账户系统

## 推荐方案

采用“主题 token 驱动的桌面播放器壳层 + Howler 播放适配层 + Iconify 图标系统”的方案。

核心理由：

- 布局、视觉、主题和播放器能力都需要被长期复用，应该沉淀成稳定的底层抽象，而不是散落在页面样式里
- `Howler.js` 足够成熟，适合当前“音乐播放器”这个问题空间，复杂度明显低于更重的音频引擎
- `Iconify` 能快速统一导航、播放器和内容操作的图标语言
- 主题改造如果只做“亮暗切换”，后续补主题色会导致样式再拆一次，不如一次把 token 体系定好

## 信息架构

### 壳层结构

- 左列：固定导航栏
- 右列：固定顶部栏 + 独立滚动主内容
- 底部：悬浮全局播放器 Dock

### 页面布局策略

- 推荐页：
  - 顶部主推荐 Hero
  - 专辑精选主模块
  - 快速入口 / 最近播放双栏
  - 氛围入口
- 我喜欢：
  - 统计摘要
  - 可播放列表
  - 当前播放状态高亮
- 个人中心：
  - 个人摘要
  - 偏好与最近活动
  - 主题与状态展示保持同一视觉语言

## 视觉语言

### 方向

- 从“整页白色玻璃”改为“深浅层叠的桌面播放器”
- 左侧导航更沉稳，右侧内容更通透
- 卡片数量减少，但层级更明确
- 强调色集中用于选中、播放、焦点和关键操作

### 亮色主题

- 背景：冷灰蓝、雾白、薄荷高光
- 导航：半透明深雾面
- 内容卡：偏浅玻璃面板
- 文字：深石墨灰

### 暗色主题

- 背景：蓝黑、深青灰、柔和环境光
- 导航：深色磨砂
- 内容卡：低透明度深玻璃
- 文字：高对比浅灰白

### 主题色

主题色不直接覆盖全部界面，而是驱动以下 token：

- 主播放键
- 进度条已播放部分
- 链接/高亮
- 选中导航
- 点赞/激活态
- 焦点环

支持两种来源：

- 预设色板
- 用户自由选择颜色值

## 组件设计

### 新增可复用基础组件

- `UiIconButton`
  - 统一图标按钮尺寸、圆角、hover、active、selected、tone
- `UiButton`
  - 文本按钮/带图标按钮统一入口
- `UiThemeToggle`
  - 主题模式切换
- `UiThemePalette`
  - 预设主题色 + 自由颜色选择
- `UiSectionCard`
  - 通用区块容器，减少重复玻璃卡片写法

### 现有组件调整

- `AppShell`
  - 改成固定导航 + 固定头部 + 内容滚动容器
- `AppSidebar`
  - 引入 Iconify 图标和更明确的激活态
- `AppTopbar`
  - 增加主题控制入口和更产品化的信息层
- `PlayerDock`
  - 更紧凑、以图标为核心的控制布局
- `PlaybackControls`
  - 纯图标操作，模式改为图标 + tooltip/aria 标签
- `VolumeControl`
  - 图标静音按钮 + 更干净的滑杆

## 主题系统设计

### 状态模型

新增主题 store，维护：

- `mode`: `light | dark | system`
- `resolvedMode`: `light | dark`
- `accent`: 当前主题色
- `presetAccentId`: 当前预设色 id，若为自定义则为空

### 实现方式

- 使用 CSS variables 作为唯一主题出口
- 在根节点写入 `data-theme` 和动态 accent 变量
- 通过 `matchMedia('(prefers-color-scheme: dark)')` 支持跟随系统
- 将原先硬编码在各组件里的颜色收拢成 token

## 播放层设计

### Why Howler

- 跨浏览器行为更稳定
- 事件模型更清晰
- 更方便控制播放、暂停、停止、seek、音量、静音和结束事件
- 适合作为当前播放器 store 的底层适配器

### 适配思路

- 保留 store 的业务语义：当前曲目、队列、模式、最近播放、收藏
- 重写 `src/lib/player/audio.ts`，对外提供与现有 store 兼容的少量控制接口
- 在测试中以 fake adapter 替代 `Howl`

## 测试策略

- 壳层测试：
  - 只有一个 `main`
  - 固定布局类名存在
  - 主题控制 UI 能渲染并驱动状态
- 主题测试：
  - 默认主题模式
  - 切换亮暗/系统
  - 选择预设主题色
  - 自定义颜色写入 token
- 播放器测试：
  - 图标按钮仍能触发播放、暂停、上一首、下一首、静音、seek
  - Howler 适配层驱动 store 状态更新

## 风险与缓解

- 风险：主题改造会触及大量样式
  - 缓解：先建立 token，再逐块替换
- 风险：Howler 与现有测试耦合方式不同
  - 缓解：通过适配层隔离，不让组件和 store 直接依赖第三方
- 风险：UI 重做范围较大
  - 缓解：先从壳层、Dock、首页三处核心区域落地，再同步其他页面
