# Aura Hi-Fi 桌面播放器全量重构设计

Date: 2026-04-09

## 摘要

基于现有 Tauri + Vue 3 + TypeScript 项目，进行一次彻底的桌面端视觉与信息架构重构。当前项目中的 3 个页面、旧壳层和旧内容组织方式不再延续，改为围绕 `Aura Hi-Fi` 这一统一产品概念，重建 5 个主页面、统一桌面壳层、统一底部播放器 Dock、统一主题系统、统一动画系统，以及基于成熟 JS/TS 开源库的播放与歌词底层能力。

本轮重构的目标不是“在现有页面上修补”，而是将项目升级成一套完整、统一、可持续扩展的桌面音乐播放器产品体验。所有页面在风格、交互、层级和组件语言上必须一致；唯一允许跨页面完全不变的核心控制区是底部播放器 Dock，并且其设计基准明确参考 `src/assets/stitch_/your_library/screen.png`。

## 目标

- 将当前 3 个页面重构为 5 个桌面主页面：
  - `Home`
  - `Explore`
  - `Playlist`
  - `Library`
  - `Player`
- 彻底替换旧页面内容与旧视觉风格，不保留旧的页面信息架构
- 建立统一的桌面端产品壳层：
  - 左侧固定导航
  - 顶部固定工具栏
  - 中间独立滚动内容区
  - 底部固定统一播放器 Dock
- 底部 Dock 在所有页面完全一致，不再允许页面之间出现不同版本
- 设计语言统一为高完成度的深色桌面播放器体验，参考 Stitch 视觉方向但以项目实现为准
- 所有图标统一使用 `Iconify`
- 所有页面布局与大量基础样式使用 `UnoCSS`
- 所有主要页面切换、卡片入场、hover 与播放器联动动画统一使用 `GSAP`
- 桌面端支持响应式布局，但暂不考虑手机端
- 明确最小宽度约束，避免界面在较小桌面窗口下被压碎
- 播放器底层优先采用成熟的 JS/TS 开源音频库
- 歌词底层优先采用成熟的 JS/TS 开源歌词解析与同步库
- 数据内容改为统一的在线模拟资源：
  - 在线图片
  - 在线音频
  - 在线或内置 LRC 歌词

## 非目标

- 不接入真实在线音乐平台、搜索 API 或后端服务
- 不实现账号体系、云同步或持久化个人资料
- 不实现移动端页面
- 不实现均衡器、波形频谱、音频可视化等高级音频能力
- 不实现复杂逐字歌词动画，第一版仅做高质量逐行同步
- 不引入大型 UI 组件框架

## 推荐方案

采用“统一设计系统 + 5 个独立页面骨架 + 全局唯一 Dock + Howler 音频适配层 + Liricle 歌词适配层 + 在线模拟内容数据层”的方案。

核心理由：

- 这次需求不是单页美化，而是整套桌面播放器产品重构，必须优先建立统一系统，再落页面
- 如果简单地把 5 张 Stitch 图逐页翻译，会导致页面之间看起来相似但代码无法维护，Dock 和交互一致性也很难守住
- `Howler.js` 适合作为浏览器 / Tauri WebView 场景下的稳定音频底层，覆盖播放、暂停、seek、音量、静音、加载与错误状态
- `Liricle` 适合作为 LRC 歌词解析与同步底层，职责明确，适合当前逐行歌词需求
- 统一组件和 token 后，`UnoCSS`、`GSAP` 和 `Iconify` 才能真正形成同一套产品语言，而不是局部使用

## 信息架构

### 一级页面

- `Home`
  - 聚合首页，对应统一后的推荐与继续播放场景
- `Explore`
  - 搜索与内容探索页，对应分类浏览、推荐发现和结果展示
- `Playlist`
  - 歌单详情页，展示某一条内容上下文及曲目表
- `Library`
  - 我的资料库页，展示收藏内容、歌单、艺人和个人收藏空间
- `Player`
  - 全屏播放器页，展示当前播放内容、歌词、控制与沉浸态视觉

### 桌面壳层

- 左侧：固定导航栏
- 顶部：固定工具栏
- 中间：页面内容滚动区
- 底部：全局统一播放器 Dock

### 页面关系

- `Home` 和 `Explore` 是内容发现入口
- `Playlist` 是从首页或探索页深入进入的内容详情页
- `Library` 是用户个人收藏与整理页
- `Player` 是当前播放内容的沉浸态呈现页
- 所有页面共享同一个播放器状态与当前播放上下文

## 页面设计

### Home

定位：应用启动后的主入口，负责“快速回到音乐状态”。

页面结构：

- 顶部 Hero 区
  - 主标题
  - 推荐专题说明
  - 主行动按钮
  - 继续播放 / 今日推荐副卡
- 中部双栏
  - 最近播放
  - 趋势推荐 / 精选专辑
- 下部扩展区
  - 氛围入口
  - 推荐歌单
  - 特色内容卡

交互要求：

- Hero 主按钮 hover 有抬升和柔和辉光
- 内容卡入场采用 stagger reveal
- 点击推荐内容时，页面主区域与底部 Dock 同步更新当前播放上下文

### Explore

定位：内容搜索与探索页。

页面结构：

- 顶部搜索区
  - 大搜索框
  - 分类筛选
  - 热门标签
- 主推荐区
  - 一张大型主题推荐卡
- 分类浏览区
  - 多组风格 / 类型入口
- 结果展示区
  - 专辑
  - 歌单
  - 艺人
  - 单曲

交互要求：

- 搜索框 focus 时出现主题色高光
- 分类卡 hover 具有轻微位移、表面提亮与阴影增强
- 默认态和搜索态切换时要平滑，不允许突然清空页面

### Playlist

定位：歌单详情与曲目消费页。

页面结构：

- 顶部 Hero 区
  - 大封面
  - 歌单标题
  - 描述
  - 创建者 / 标签 / 统计信息
  - 播放与收藏按钮
- 曲目操作区
  - 主播放
  - 收藏
  - 更多操作
- 曲目表
  - 当前播放高亮
  - 行内播放入口
  - 行内收藏入口
- 附加区
  - 相似歌单或相关推荐

交互要求：

- Hero 封面 hover 有轻微 zoom
- 曲目行 hover 显示背景抬升与操作态
- 当前播放曲目行与底部 Dock 保持联动高亮

### Library

定位：个人音乐收藏馆。

页面结构：

- 顶部标题区
  - `Your Library` 大标题

  - 排序 / 过滤按钮

  - 一张大型 `Liked Songs` 焦点卡

  - 常听合集

交互要求：

- 小卡片 hover 统一上浮

### Player

定位：沉浸式全屏播放器页。

页面结构：

- 主视觉区

  - 曲名
  - 艺人

  - 标签
- 歌词区
  - 同步歌词列表
  - 当前行居中高亮

  - 大型播放控制
  - 时间轴

交互要求：

- 歌词高亮必须跟随播放时间变化

## 统一视觉系统

### 总体方向

- 深色基底，不使用生硬纯黑
- 通过大标题、留白和层级建立高级感
- 不依赖 1px 边框切割区块
- 使用背景层次、面板表面变化和局部高光区分结构
- 强调色只用于关键操作、焦点与播放状态

### 视觉层级

- 场景层
  - 页面背景、光斑、整体环境氛围
- 面板层
  - 侧栏、顶栏、内容卡、Dock
- 交互层
  - hover、active、focus、当前播放、选中态

### 字体与排版

- 大标题与展示级文字使用更强烈的层级和更紧凑的字距
- 正文与说明文字保证在桌面端长时间阅读下的清晰度
- 信息密度高的区域通过字号、颜色和留白区分主次

### 主题系统

支持：

- `light`
- `dark`
- `system`
- 预设强调色
- 自定义主题色

实现原则：

- 以 CSS Variables 为唯一主题出口
- 根节点使用 `data-theme` / `data-theme-resolved`
- 强调色通过动态变量驱动：
  - 主播放按钮
  - 进度条已播放部分
  - 选中导航
  - focus ring
  - 激活态按钮

## 底部播放器 Dock

### 核心约束

- 全站仅允许一个版本的底部 Dock
- 所有页面完全复用同一组件与同一视觉结构
- 设计风格明确以 `src/assets/stitch_/your_library/screen.png` 为参考基准

### 结构

- 左侧
  - 当前封面
  - 曲名
  - 艺人 / 专辑
  - 收藏或状态信息
- 中间
  - 上一首
  - 播放 / 暂停
  - 下一首
  - 播放模式
  - 进度条
- 右侧
  - 音量
  - 队列入口
  - 跳转全屏播放器入口
  - 补充状态

### 视觉规则

- 比当前实现更扁、更稳定、更偏横向控制台
- 使用深色磨砂表面、柔和高光和底部环境辉光
- 播放键是唯一权重最高的主控件
- 进度条 hover 时出现把手和高亮，但常态不过度抢视觉

### 交互规则

- 切歌时封面、标题与副标题同步过渡
- hover 统一为轻抬、亮度上升、阴影增强
- 按下反馈统一为轻微缩放
- 不允许某一页面给 Dock 单独增加不同风格的控制区

## 动画系统

### 技术要求

- 所有关键动画统一使用 `GSAP`
- 不混用多套动画语义

### 动画分类

- 页面切换动画
  - 顶层区块分层 reveal
- 卡片 / 列表动画
  - stagger 入场
  - hover 上浮
  - 封面轻微 tilt 或 zoom
- 播放器联动动画
  - 切歌
  - 当前播放项高亮
  - 歌词当前行跟随

### 动画节奏

- 大区块入场：`0.55s - 0.8s`
- 列表 stagger：`0.04s - 0.1s`
- hover：`0.18s - 0.24s`
- press：`0.12s - 0.18s`

### 动画原则

- 动画必须服务于信息表达与产品完成度
- 页面之间的动画手感保持一致
- 不做无意义炫技动画

## 技术实现策略

### UnoCSS

`UnoCSS` 负责：

- 栅格与布局
- 间距与尺寸
- 排版类
- 响应式断点
- 一部分轻量状态类

复杂组件皮肤、渐变、玻璃质感和长逻辑样式仍通过组件样式和主题 token 管理，避免模板中堆积不可维护的超长 class。

### Iconify

- 所有导航图标
- 所有播放器控制图标
- 所有状态图标
- 所有操作图标

统一由 `Iconify` 提供，不保留旧的文本替代或混杂图标来源。

### 音频底层

推荐使用 `Howler.js`。

职责：

- 加载音频
- 播放 / 暂停
- seek
- 音量
- 静音
- 播放结束事件
- 错误处理
- 持续时间与当前时间同步

实现原则：

- 页面和组件不直接依赖 `Howler`
- 通过 `audio adapter` 暴露统一接口
- `player store` 只依赖 adapter，不依赖第三方具体实现

### 歌词底层

推荐使用 `Liricle`。

职责：

- 解析 LRC 文本
- 根据当前播放时间定位当前行
- 支持切歌重载歌词
- 支持基础 offset 校正

实现原则：

- 歌词 UI 不直接依赖第三方库
- 通过 `lyrics adapter` 或 `lyrics store` 提供：
  - 当前歌词数组
  - 当前高亮行索引
  - 加载状态
  - 无歌词状态

第一版只实现高质量逐行同步，不实现逐字卡拉 OK。

## 数据与资源策略

### 在线资源

- 封面图统一使用在线图片 URL
- 音频统一使用免费可访问的在线音频 URL
- 歌词优先使用在线或内置 LRC 资源

### 数据模型

每首曲目至少包含：

- `id`
- `title`
- `artist`
- `album`
- `coverSrc`
- `audioSrc`
- `lyrics`
- `duration`
- `moods`
- `tags`

### 组织原则

- 页面不直接散写资源 URL
- 所有内容由统一数据层提供
- 推荐内容、歌单、收藏夹、艺人等都从同一套模拟内容源派生

## 响应式与最小尺寸

### 范围

- 本轮仅支持桌面端
- 不提供手机端适配

### 最小尺寸约束

- 应用最小宽度：`1280px`
- 推荐工作宽度：`1360px - 1720px`

### 缩放原则

- 小桌面下减少列数、缩减空白和边距
- 不压缩到让卡片与控件失真
- 小于最小宽度时优先保持结构完整，可通过容器最小宽度与滚动规避塌陷

## 测试策略

### 壳层与路由

- 5 个主页面路由存在并可渲染
- 左侧导航、顶部栏和统一 Dock 始终存在
- 内容区滚动结构正确

### 播放器

- 播放 / 暂停 / 上一首 / 下一首 / seek / 音量 / 静音 可正常驱动 store
- 切歌会同步更新 Dock、页面高亮和 Player 页面内容

### 歌词

- 有歌词时能正确高亮当前行
- 无歌词时能渲染空状态
- 切歌后能重置并加载新的歌词上下文

### 页面结构

- `Home`、`Explore`、`Playlist`、`Library`、`Player` 的关键区块存在
- `Library` 页的大焦点卡与 `Player` 页歌词区可被稳定定位

### 响应式

- 在桌面最小宽度下主结构不塌陷
- Dock 仍保持完整结构

## 风险与缓解

- 风险：这次属于整站重写，改动面大
  - 缓解：先建立统一壳层、设计 token、Dock 和内容数据层，再逐页替换
- 风险：在线音频与图片资源稳定性可能波动
  - 缓解：优先选择稳定公共资源，并为音频加载失败提供清晰状态
- 风险：第三方音频与歌词库会改变现有测试方式
  - 缓解：通过 adapter 隔离依赖，测试中以 fake adapter / mock data 替代真实实现
- 风险：页面容易各自好看但整体不统一
  - 缓解：先锁定统一设计系统、统一动画节奏和唯一 Dock，再开发页面

## 实施边界

本轮允许重写现有页面、主要壳层组件和数据组织方式；不要求保留当前旧页面的文案、模块结构和视觉表达。唯一需要保留的是项目的技术栈基础、可测试性和整体可维护性。
