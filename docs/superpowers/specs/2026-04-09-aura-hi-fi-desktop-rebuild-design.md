# Aura Hi-Fi 桌面播放器全量重构设计

Date: 2026-04-09

## 摘要

基于现有 Tauri + Vue 3 + TypeScript 项目，进行一次彻底的桌面端视觉与信息架构重构。当前项目中的 3 个页面、旧壳层和旧内容组织方式不再延续，改为围绕 `Aura Hi-Fi` 这一统一产品概念，重建 5 个主页面、统一桌面壳层、统一底部播放器 Dock、统一主题系统、统一动画系统，以及基于成熟 JS/TS 开源库的播放与歌词底层能力。

本轮重构的目标不是“在现有页面上修补”，而是将项目升级成一套完整、统一、可持续扩展的桌面音乐播放器产品体验。所有页面在风格、交互、层级和组件语言上必须一致；唯一允许跨页面完全不变的核心控制区是底部播放器 Dock，并且其设计基准明确参考 `src/assets/stitch_/your_library/screen.png`。

## 交付范围

本轮交付是单个产品重构项目，而不是多个彼此独立的子系统项目。实现计划允许拆成多个阶段任务，但所有任务共同服务于一个统一交付物：

- 一个重构后的桌面播放器应用
- 一套统一的桌面端设计系统
- 5 个新的主页面
- 一套统一的播放与歌词能力

为了避免后续计划拆分失焦，本轮里程碑固定为：

- 里程碑 1：建立统一桌面壳层、路由、设计 token 和唯一 Dock
- 里程碑 2：建立在线模拟内容数据层、音频 adapter、歌词 adapter
- 里程碑 3：完成 5 个页面主体结构与主要交互
- 里程碑 4：补齐动画、hover、桌面端响应式与测试

## 目标

- 将当前 3 个页面重构为 5 个桌面主页面：
  - `Home`
  - `Explore`
  - `Playlist`
  - `Library`
  - `Player`
- 彻底替换旧页面内容与旧视觉风格，不保留旧的页面信息架构
- 建立统一的桌面端产品壳层：
  - 左侧固定导航
  - 顶部固定工具栏
  - 中间独立滚动内容区
  - 底部固定统一播放器 Dock
- 底部 Dock 在所有页面完全一致，不再允许页面之间出现不同版本
- 设计语言统一为高完成度的深色桌面播放器体验，参考 Stitch 视觉方向但以项目实现为准
- 所有图标统一使用 `Iconify`
- 所有页面布局与大量基础样式使用 `UnoCSS`
- 所有主要页面切换、卡片入场、hover 与播放器联动动画统一使用 `GSAP`
- 所有面向用户的页面文案统一使用中文：
  - 页面标题
  - 按钮文字
  - 导航标签
  - 提示语
  - 空状态
  - 播放状态
- 桌面端支持响应式布局，但暂不考虑手机端
- 明确最小宽度约束，避免界面在较小桌面窗口下被压碎
- 播放器底层优先采用成熟的 JS/TS 开源音频库
- 歌词底层优先采用成熟的 JS/TS 开源歌词解析与同步库
- 数据内容改为统一的在线模拟资源：
  - 在线图片
  - 在线音频
  - 在线或内置 LRC 歌词

## 非目标

- 不接入真实在线音乐平台、搜索 API 或后端服务
- 不实现账号体系、云同步或持久化个人资料
- 不实现移动端页面
- 不实现均衡器、波形频谱、音频可视化等高级音频能力
- 不实现复杂逐字歌词动画，第一版仅做高质量逐行同步
- 不引入大型 UI 组件框架
- 不在本轮交付亮色主题、跟随系统主题和主题色设置面板
- 不保留 Stitch 设计稿中的英文 UI 文案作为最终交付内容

## 推荐方案

采用“统一设计系统 + 5 个独立页面骨架 + 全局唯一 Dock + Howler 音频适配层 + Liricle 歌词适配层 + 在线模拟内容数据层”的方案。

核心理由：

- 这次需求不是单页美化，而是整套桌面播放器产品重构，必须优先建立统一系统，再落页面
- 如果简单地把 5 张 Stitch 图逐页翻译，会导致页面之间看起来相似但代码无法维护，Dock 和交互一致性也很难守住
- `Howler.js` 适合作为浏览器 / Tauri WebView 场景下的稳定音频底层，覆盖播放、暂停、seek、音量、静音、加载与错误状态
- `Liricle` 适合作为 LRC 歌词解析与同步底层，职责明确，适合当前逐行歌词需求
- 统一组件和 token 后，`UnoCSS`、`GSAP` 和 `Iconify` 才能真正形成同一套产品语言，而不是局部使用

## 信息架构

### 一级页面

- `Home`
  - 聚合首页，对应统一后的推荐与继续播放场景
- `Explore`
  - 搜索与内容探索页，对应分类浏览、推荐发现和结果展示
- `Playlist`
  - 歌单详情页，展示某一条内容上下文及曲目表
- `Library`
  - 我的资料库页，展示收藏内容、歌单、艺人和个人收藏空间
- `Player`
  - 全屏播放器页，展示当前播放内容、歌词、控制与沉浸态视觉

### 桌面壳层

- 左侧：固定导航栏
- 顶部：固定工具栏
- 中间：页面内容滚动区
- 底部：全局统一播放器 Dock

### 导航与顶部栏最小功能清单

左侧导航固定顺序：

- `Home`
- `Explore`
- `Playlist`
- `Library`
- `Player`

导航状态规则：

- 当前路由：高亮、带强调色激活态
- 可跳转但未选中：默认态 + hover 抬升
- 本轮不设计禁用导航项

顶部工具栏最小功能清单：

- 当前页面标题
- 搜索入口
- 至少一个状态动作按钮
- 用户头像或个人入口占位

顶部栏不承载播放主控制，不与底部 Dock 重复职责。

搜索入口规则：

- `Explore` 页面顶部显示可直接输入的搜索框
- 其他页面顶部显示“进入探索”搜索入口按钮
- 点击其他页面的搜索入口后，路由跳转到 `Explore` 并自动聚焦搜索框

状态动作按钮规则：

- 本轮固定为“应用状态”按钮
- 点击后可打开轻量状态面板或占位 popover
- 该按钮不是播放控制入口，不承载主题切换、通知中心或复杂设置

### 页面关系

- `Home` 和 `Explore` 是内容发现入口
- `Playlist` 是从首页或探索页深入进入的内容详情页
- `Library` 是用户个人收藏与整理页
- `Player` 是当前播放内容的沉浸态呈现页
- 所有页面共享同一个播放器状态与当前播放上下文

`Player` 页定义补充：

- 这里的“全屏播放器页”指的是应用内沉浸式页面，不是系统级原生全屏窗口
- `Player` 页仍然保留左侧导航、顶部栏和底部 Dock
- 进入 `Player` 页后，主内容区的视觉权重显著提升，壳层弱化但不消失
- 因此测试策略中“左侧导航、顶部栏和统一 Dock 始终存在”依然成立

### 路由策略

- `Home`：`/`
- `Explore`：`/explore`
- `Playlist`：
  - 一级入口路由：`/playlist`
  - 具体歌单深链：`/playlist/:playlistId`
- `Library`：`/library`
- `Player`：`/player`

`Playlist` 入口规则：

- 左侧导航进入 `/playlist` 时，展示一张固定的默认主打歌单
- 从 `Home` 或 `Explore` 中点击某张歌单卡片时，进入 `/playlist/:playlistId`
- 本轮不做“记住上次访问歌单并自动恢复”为导航默认行为

## 统一视觉系统

### 总体方向

- 深色基底，不使用生硬纯黑
- 通过大标题、留白和层级建立高级感
- 不依赖 1px 边框切割区块
- 使用背景层次、面板表面变化和局部高光区分结构
- 强调色只用于关键操作、焦点与播放状态

### 视觉层级

- 场景层
  - 页面背景、光斑、整体环境氛围
- 面板层
  - 侧栏、顶栏、内容卡、Dock
- 交互层
  - hover、active、focus、当前播放、选中态

### 字体与排版

- 大标题与展示级文字使用更强烈的层级和更紧凑的字距
- 正文与说明文字保证在桌面端长时间阅读下的清晰度
- 信息密度高的区域通过字号、颜色和留白区分主次

### 文案语言规则

- 最终用户可见文案统一为中文
- 设计稿中的英文只作为结构参考，不直接进入最终页面
- 文档中的 `Home`、`Explore`、`Playlist`、`Library`、`Player` 仅作为内部页面标识与实现命名，不作为最终 UI 文案
- 包括但不限于以下内容：
  - 顶部标题
  - 导航项
  - 按钮
  - 输入框 placeholder
  - 状态提示
  - 歌词空状态
  - 队列面板标题
  - 收藏与播放相关提示

### 主题系统

本轮仅交付一套完整的深色主题，不交付亮色、跟随系统和自定义主题色 UI。

实现原则：

- 以 CSS Variables 为唯一主题出口
- 预留强调色 token，但不提供用户配置面板
- 强调色驱动：
  - 主播放按钮
  - 进度条已播放部分
  - 选中导航
  - focus ring
  - 激活态按钮

补充说明：

- 若现有代码中已经存在主题 store，本轮允许简化、收口或移除与本轮交付无关的主题能力
- 本轮追求“统一完成度”，而不是“多主题覆盖面”

## 底部播放器 Dock

### 核心约束

- 全站仅允许一个版本的底部 Dock
- 所有页面完全复用同一组件与同一视觉结构
- 设计风格明确以 `src/assets/stitch_/your_library/screen.png` 为参考基准
- 若其他页面视觉表达与 Dock 风格参考出现冲突，以 `src/assets/stitch_/your_library/screen.png` 的 Dock 语义与质感为最高优先级

### 结构

- 左侧
  - 当前封面
  - 曲名
  - 艺人 / 专辑
  - 收藏或状态信息
- 中间
  - 上一首
  - 播放 / 暂停
  - 下一首
  - 播放模式
  - 进度条
- 右侧
  - 音量
  - 队列入口
  - 跳转全屏播放器入口
  - 补充状态

### 最小可交付语义

播放模式仅交付 3 种状态：

- 顺序播放
- 单曲循环
- 随机播放

播放模式规则：

- 通过 Dock 中的模式按钮循环切换
- 模式状态写入 `player store`
- 所有页面共享同一播放模式状态
- 顺序播放：
  - `next` 跳到队列中的下一首
  - `previous` 回到当前队列中的上一首
  - 队列播放到最后一首后停止播放并停留在最后一首
- 单曲循环：
  - 当前曲目结束后重新播放当前曲目
  - `next` 与 `previous` 仍可手动切换到队列中的相邻曲目
  - 手动切歌后，新曲目继续处于单曲循环模式
- 随机播放：
  - 基于当前播放上下文生成一份稳定的随机顺序队列
  - 这份随机顺序在当前上下文不变时保持稳定
  - `next` 与 `previous` 在这份随机顺序队列中移动
  - 随机队列播到最后一首后停止播放并停留在最后一首

队列入口规则：

- 本轮交付的是“查看 upcoming 队列”的只读入口，不做拖拽编辑、删除、插队
- 队列来源于当前播放上下文：
  - 当前歌单
  - 当前推荐列表
  - 当前专辑或页面触发的播放集合
- 队列入口打开后展示：
  - 当前播放曲目
  - 下一首列表
  - 总队列长度
- 队列可以点击切歌，但不支持改写队列结构

收藏状态规则：

- 收藏仅为当前会话内状态
- 本轮不接入本地持久化、数据库或云同步
- 应用刷新或重启后，收藏状态允许回到默认模拟数据
- 在当前会话内，收藏状态需要跨页面即时同步：
  - `Playlist`
  - `Library`
  - `Dock`
  - 其他出现收藏按钮的内容卡

### 视觉规则

- 比当前实现更扁、更稳定、更偏横向控制台
- 使用深色磨砂表面、柔和高光和底部环境辉光
- 播放键是唯一权重最高的主控件
- 进度条 hover 时出现把手和高亮，但常态不过度抢视觉

### 交互规则

- 切歌时封面、标题与副标题同步过渡
- hover 统一为轻抬、亮度上升、阴影增强
- 按下反馈统一为轻微缩放
- 不允许某一页面给 Dock 单独增加不同风格的控制区
- Dock 进入 `Player` 页的主触发方式为：
  - 点击右侧“全屏播放器”图标按钮进行路由跳转
- 从 `Player` 页返回的主方式为：
  - 使用左侧导航跳回其他页面
  - 或使用浏览器 / 路由返回

## 动画系统

### 技术要求

- 所有关键动画统一使用 `GSAP`
- 不混用多套动画语义

### 动画分类

- 页面切换动画
  - 顶层区块分层 reveal
- 卡片 / 列表动画
  - stagger 入场
  - hover 上浮
  - 封面轻微 tilt 或 zoom
- 播放器联动动画
  - 切歌
  - 当前播放项高亮
  - 歌词当前行跟随

### 动画节奏

- 大区块入场：`0.55s - 0.8s`
- 列表 stagger：`0.04s - 0.1s`
- hover：`0.18s - 0.24s`
- press：`0.12s - 0.18s`

### 动画原则

- 动画必须服务于信息表达与产品完成度
- 页面之间的动画手感保持一致
- 不做无意义炫技动画

## 技术实现策略

### UnoCSS

`UnoCSS` 负责：

- 栅格与布局
- 间距与尺寸
- 排版类
- 响应式断点
- 一部分轻量状态类

复杂组件皮肤、渐变、玻璃质感和长逻辑样式仍通过组件样式和主题 token 管理，避免模板中堆积不可维护的超长 class。

### Iconify

- 所有导航图标
- 所有播放器控制图标
- 所有状态图标
- 所有操作图标

统一由 `Iconify` 提供，不保留旧的文本替代或混杂图标来源。

### 音频底层

推荐使用 `Howler.js`。

职责：

- 加载音频
- 播放 / 暂停
- seek
- 音量
- 静音
- 播放结束事件
- 错误处理
- 持续时间与当前时间同步

实现原则：

- 页面和组件不直接依赖 `Howler`
- 通过 `audio adapter` 暴露统一接口
- `player store` 只依赖 adapter，不依赖第三方具体实现
- 曲目 `duration` 由模拟数据层提供静态值，作为 UI 首屏展示基准
- 音频资源加载成功后，若真实时长可获取，则允许回填 store 中的运行时时长
- 在真实时长尚未返回前，UI 展示静态模拟时长，不进入“未知时长”空白态

### 歌词底层

推荐使用 `Liricle`。

职责：

- 解析 LRC 文本
- 根据当前播放时间定位当前行
- 支持切歌重载歌词
- 支持基础 offset 校正

实现原则：

- 歌词 UI 不直接依赖第三方库
- 通过 `lyrics adapter` 或 `lyrics store` 提供：
  - 当前歌词数组
  - 当前高亮行索引
  - 加载状态
  - 无歌词状态

第一版只实现高质量逐行同步，不实现逐字卡拉 OK。

## 数据与资源策略

### 在线资源

- 封面图统一使用在线图片 URL
- 音频统一使用免费可访问的在线音频 URL
- 歌词优先使用在线或内置 LRC 资源

### 数据模型

每首曲目至少包含：

- `id`
- `title`
- `artist`
- `album`
- `coverSrc`
- `audioSrc`
- `lyrics`
- `duration`
- `moods`
- `tags`

### 组织原则

- 页面不直接散写资源 URL
- 所有内容由统一数据层提供
- 推荐内容、歌单、收藏夹、艺人等都从同一套模拟内容源派生
- 实现计划中必须单列“在线 mock 资源包选型与验收”任务，先确认可稳定访问的图片、音频和歌词资源，再进入页面联调
- 搜索仅针对本地 mock 数据执行客户端过滤
- 搜索覆盖字段至少包括：
  - 曲名
  - 艺人名
  - 专辑名
  - 歌单名
  - 标签
- 搜索交互采用轻量 debounce，目标值为 `150ms`
- 需要明确展示：
  - 初始默认探索态
  - 有输入但无结果的空状态
  - 有结果时的分组结果态

## 响应式与最小尺寸

### 范围

- 本轮仅支持桌面端
- 不提供手机端适配

### 最小尺寸约束

- 内容布局最小宽度：`1280px`
- 推荐工作宽度：`1360px - 1720px`

### 缩放原则

- 小桌面下减少列数、缩减空白和边距
- 不压缩到让卡片与控件失真
- 本轮不强制修改 Tauri 窗口最小宽度配置
- 若窗口宽度小于 `1280px`，通过内容容器 `min-width` 与横向滚动优先保证结构完整

## 测试策略

### 壳层与路由

- 5 个主页面路由存在并可渲染
- 左侧导航、顶部栏和统一 Dock 始终存在
- 内容区滚动结构正确

### 播放器

## 风险与缓解

- 风险：这次属于整站重写，改动面大
  - 缓解：先建立统一壳层、设计 token、Dock 和内容数据层，再逐页替换
- 风险：在线音频与图片资源稳定性可能波动
  - 缓解：优先选择稳定公共资源，并为音频加载失败提供清晰状态
- 风险：第三方音频与歌词库会改变现有测试方式
  - 缓解：通过 adapter 隔离依赖，测试中以 fake adapter / mock data 替代真实实现
- 风险：页面容易各自好看但整体不统一
  - 缓解：先锁定统一设计系统、统一动画节奏和唯一 Dock，再开发页面

## 实施边界

本轮允许重写现有页面、主要壳层组件和数据组织方式；不要求保留当前旧页面的文案、模块结构和视觉表达。唯一需要保留的是项目的技术栈基础、可测试性和整体可维护性。
