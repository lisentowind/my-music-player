# 液态玻璃桌面音乐播放器设计

Date: 2026-03-23

## 摘要

将当前 `Tauri + Vue 3 + TypeScript` starter 改造成一套高保真桌面音乐播放器应用，目标是同时具备可展示的高级 UI 质感与可运行的核心播放交互。产品包含三个核心页面：推荐、我喜欢、个人中心；并提供一个全局悬浮底部播放 Dock，支持真实 mock 音频播放、切歌、进度控制、音量控制与收藏联动。

## 目标

- 将现有 starter 首页重构为完整的桌面音乐播放器产品壳
- 提供三个核心页面：
  - 推荐
  - 我喜欢
  - 个人中心
- 构建全局播放器状态层，支持真实本地 mock 音频播放
- 统一视觉语言为 `macOS / Apple Music / Liquid Glass`
- 让 Web 与 Tauri 双运行态都能稳定展示与播放
- 保持当前工程基线可维护，不引入与本阶段目标无关的重量级依赖

## 非目标

- 接入真实后端接口或在线音乐服务
- 实现本地文件夹扫描、媒体库导入、歌词解析、均衡器等高级功能
- 引入数据库持久化或账号系统
- 实现复杂动效引擎或 3D 特效

## 推荐方案

采用“高保真产品壳 + 本地 mock 数据 + 真实音频播放”的方式推进。

核心思路：

- UI 层直接围绕桌面播放器体验重做，而不是继续扩展 starter 占位页
- 数据层使用本地 mock 媒体库，保证内容可控、视觉完整
- 播放层使用浏览器原生 `HTMLAudioElement` 承载音频播放能力，而不是提前引入原生音频插件
- 全局状态统一由 Pinia 管理，所有页面与底部 Dock 共用同一套播放上下文

这样能在当前阶段同时满足三件事：

1. 视觉足够完整，接近作品集级桌面应用
2. 交互不是假动作，播放链路真实可用
3. 后续若接入真实媒体库或 Tauri 文件能力，不需要推倒 UI 与状态层

## 备选方案

### 方案 A：只做高保真静态原型

优点：出效果最快。

缺点：播放、进度、切歌都只是演示态，不符合当前“半真实应用”的目标。

### 方案 B：直接做本地文件扫描型播放器

优点：更接近正式产品。

缺点：范围迅速膨胀，容易把本轮工作重心从 UI 质量和交互完成度转移到桌面文件处理与权限问题上。

### 方案 C：高保真产品壳 + mock 媒体库 + 真实音频播放

推荐。范围和效果平衡最好，适合作为当前阶段的第一版产品基线。

## 信息架构

应用采用三段式桌面布局：

- 左侧：玻璃拟态侧边栏导航
- 中间：主内容区
- 底部：全局悬浮播放 Dock

### 页面结构

#### 推荐

承担内容发现与视觉展示职责，包含：

- 顶部欢迎区与今日摘要
- 推荐歌单 / 专辑卡片网格
- 最近播放
- 为你精选或氛围分类卡片

#### 我喜欢

承担可操作的媒体列表职责，包含：

- 收藏歌曲统计
- 可播放的歌曲表格
- 当前播放高亮
- 收藏、时长、专辑、歌手等信息

#### 个人中心

承担用户展示与轻分析职责，包含：

- 用户卡片
- 听歌时长、收藏数量、连续收听等数据
- 常听风格
- 最近活动或最近加入喜欢列表

## 架构设计

### 路由层

将现有 `home/about` 路由重构为产品化三页：

- `/` -> 推荐
- `/liked` -> 我喜欢
- `/profile` -> 个人中心

继续使用 `createWebHashHistory()`，保证 Tauri 静态宿主兼容性不退化。

### 数据层

新增一份本地 mock 媒体数据模块，包含：

- 歌曲列表
- 专辑 / 歌单封面
- 歌手信息
- 分类标签
- 默认收藏状态
- 本地音频资源路径

数据结构需要足够统一，便于同时驱动卡片视图、列表视图、个人页统计与底部 Dock。

### 播放层

新增全局播放器 store，统一管理：

- 当前播放歌曲
- 当前播放列表
- 当前索引
- 是否播放中
- 当前播放时间
- 总时长
- 音量
- 静音状态
- 播放模式
- 收藏状态

音频实例由一处集中创建与绑定，避免多个组件分别控制 `audio` 导致状态分裂。

### 组件层

组件拆分遵循“展示组件”和“播放器/业务组件”分离原则：

- 布局组件：应用外壳、侧边栏、页头、玻璃卡片容器
- 内容组件：专辑卡、推荐块、统计卡、歌曲表格
- 播放器组件：底部 Dock、进度条、播放按钮组、音量控件、当前歌曲信息

## 视觉语言

整体风格对齐以下关键词：

- 冷淡
- 通透
- 高级
- 苹果感
- 空气感
- 空间层级明确

不追求霓虹感或赛博灯效，避免“发绿、发脏、塑料感”的蓝绿污染。

### 背景

背景采用低饱和冰蓝、暖白、极淡金色混合的空气感渐变，强调环境氛围，而不是高对比色块。

### 玻璃材质

全局卡片与 Dock 以白色半透明玻璃为主：

- 主体：`rgba(255,255,255,0.65 ~ 0.78)`
- 模糊：`blur(24px ~ 32px)`
- 饱和度增强：轻度 `saturate()`
- 边框：极细半透明白边
- 阴影：柔和浅阴影，不使用厚重黑影

### 图标与文字

- 图标统一为线性风格
- 颜色采用低饱和深灰
- 标题更厚重，辅助信息更轻更淡
- 整体文案使用中文产品感信息架构

## Dock 专项设计

底部 Dock 是本次改造的核心。

### 当前问题

- 蓝绿渐变过重，显脏且偏荧光
- 中央按钮发光感过强，接近电竞灯效
- Dock 与背景层级不够明确
- 玻璃质感更像彩色玻璃，不像液态玻璃

### 目标效果

Dock 应呈现“悬浮在空气中的玻璃条”：

- 白色液态玻璃主导
- 有真实光感，但没有霓虹污染
- 层级清楚，像悬浮于主内容区上方
- 播放按钮像真实物理按钮，而不是发光按钮

### 设计策略

#### Dock 主体

- 去掉重彩蓝绿渐变
- 使用半透明白色玻璃背景
- 使用更强的背景模糊与轻边框
- 增加更明显的悬浮阴影和边缘高光

#### 环境光层

在 Dock 背后增加极淡径向氛围光，用于营造“苹果发布会感”，但不直接污染主体颜色。

#### 播放按钮

- 改为白色或极淡冷白渐变按钮
- 保留物理按键感和轻微 hover 放大
- 播放中用极弱脉冲缩放提示状态
- 去掉大面积外发光

#### 进度条

- 轨道为玻璃感浅白半透明
- 进度为清透蓝色渐变
- 滑块采用白色小圆点，强调细腻的物理感

#### 专辑封面与控制图标

- 封面增加轻边框和柔和投影
- 图标统一线性深灰风格
- hover 只做轻度提亮，不做高饱和变色

## 页面行为设计

### 推荐页

- 点击卡片或推荐歌曲可立即切歌
- 推荐区会显示当前播放歌曲关联的状态
- 最近播放可复用当前播放器历史或 mock 最近项

### 我喜欢页

- 歌曲表格支持点击整行播放
- 可直接切换喜欢状态
- 当前播放歌曲高亮，并显示播放中提示

### 个人中心页

- 各项统计来自 mock 数据与全局收藏 / 播放状态推导
- 最近活动与常听内容可根据当前 mock 数据计算生成

## 交互行为设计

### 播放控制

支持：

- 播放 / 暂停
- 上一首 / 下一首
- 点击歌曲播放
- 拖动进度条跳转
- 自动播放下一首
- 顺序播放 / 单曲循环 / 列表循环

### 音量控制

支持：

- 音量滑块
- 静音切换
- UI 即时反馈

### 收藏联动

收藏状态由全局 store 管理，推荐页、我喜欢页、Dock 内按钮需要保持一致。

## 响应式策略

本项目主要面向桌面端，但仍需要保证窄宽度窗口下不崩坏。

- 主布局以宽屏桌面优先
- 中等宽度时允许内容区重排
- 小宽度时侧边栏收紧、Dock 分段压缩

目标不是移动端 App 化，而是在桌面窗口缩放时保持优雅。

## 文件组织建议

建议新增或重构以下模块：

```text
src/
  components/
    chrome/
    dock/
    music/
  data/
  stores/
  views/
```

职责建议：

- `components/chrome`: 应用壳、侧边栏、顶栏等框架组件
- `components/dock`: 播放 Dock 及其子组件
- `components/music`: 专辑卡、歌曲列表、统计卡等业务组件
- `data`: mock 歌曲、封面、用户信息与统计派生输入
- `stores`: 全局播放器与偏好状态

## 验收标准

当以下条件成立时，本轮改造视为完成：

- 三个核心页面都已替换掉 starter 占位内容
- 底部 Dock 全局常驻，并符合液态玻璃方向
- 至少数首本地 mock 音频可真实播放
- 播放 / 暂停 / 上一首 / 下一首 / 进度拖拽 / 音量调整可用
- 推荐页与我喜欢页可以驱动全局播放器
- 收藏状态在多个页面之间保持同步
- `pnpm lint`
- `pnpm typecheck`
- `pnpm build:web`
- `pnpm build`

## 风险与缓解

### 风险：没有现成音频资源

缓解：引入少量体积可控的 mock 音频文件，仅用于播放器演示。

### 风险：高保真玻璃样式导致样式复杂度膨胀

缓解：先建立统一设计 token 和玻璃卡片基础样式，避免每个组件单独手搓一套视觉规则。

### 风险：音频实例与 UI 状态不同步

缓解：所有音频事件绑定集中在播放器 store 或其专用控制模块中，组件只消费状态与动作。

### 风险：桌面小窗口下布局拥挤

缓解：优先设计桌面断点下的分段收缩规则，特别处理 Dock 和歌曲表格区域。

## 已确认决策

- 页面范围：推荐 / 我喜欢 / 个人中心，三页全做
- 文案方向：中文产品感
- 交互层级：半真实应用
- 播放能力：真实 mock 音频播放
- 视觉风格：macOS / Apple Music / Liquid Glass
- Dock 方向：白色液态玻璃主导，去霓虹、去荧光、去塑料感
