# 播放器外壳与主题系统升级设计

Date: 2026-03-24

## 摘要

在保留当前 `macOS / Liquid Glass` 视觉方向的前提下，对现有桌面音乐播放器做一轮结构化升级。本次升级不重做产品信息架构，而是在现有推荐、我喜欢、个人中心三页与全局底部 Dock 的基础上，补齐四类关键能力：

- 亮色 / 暗色切换
- 预设主题色 + 自定义主色
- 全站统一接入 Iconify 图标并恢复真实按钮语义
- 修复桌面布局，使左侧与右侧内容区同高，右侧独立滚动，底部播放器拥有真实进度控制体验

目标是让当前项目从“视觉完成度较高的播放器雏形”升级为“桌面结构稳定、交互一致、外观可配置”的可持续产品壳。

## 已确认决策

- 保持现有风格：`macOS` 玻璃风，不转向扁平化或霓虹风
- 全局文案：中文
- 暗色 / 亮色切换入口：顶部栏右侧
- 主题色策略：预设主题色 + 自定义主色
- 图标策略：统一使用 `Iconify`
- 交互策略：不再使用纯文字替代按钮，操作位必须使用真实按钮；必要处可采用“图标 + 文字”组合
- 布局目标：左侧固定高度、元素居中；右侧与左侧同高，内容超出时在右侧内部滚动
- 播放器目标：底部 Dock 保持现有玻璃风外形，但进度条和控制交互需要更真实

## 目标

- 建立可扩展的外观系统，为亮色、暗色、主题色切换提供统一变量入口
- 在不破坏现有玻璃风质感的前提下，完成全局主题联动
- 让桌面端布局稳定在单视窗内，避免右侧无限增长导致左侧错位
- 将导航、排序、喜欢、播放、音量、主题切换等交互统一为真实按钮
- 用 Iconify 替换当前大部分纯文字操作控件
- 升级底部播放器的操作反馈，使进度条、模式、音量、按钮样式和主题系统保持一致

## 非目标

- 不重做页面信息架构
- 不新增在线音乐服务、账号系统或后端接口
- 不做完整高级设置页
- 不做跨会话播放历史系统
- 不引入额外 UI 组件库

## 方案对比

### 方案 A：局部补丁式修改

逐个组件打补丁，加暗黑模式类名、替换几个按钮、修正少量布局样式。

优点：

- 实现快
- 改动面小

缺点：

- 主题逻辑会分散在多个组件里
- 后续继续加外观设置会迅速失控
- 布局问题容易只被“压住”，没有真正建立稳定壳层

### 方案 B：结构化升级，保留现有产品壳

新增独立的外观状态层，统一主题变量；重构应用外壳布局；为按钮和图标建立一致规则；升级底部播放器交互。

优点：

- 能解决当前的根问题
- 与现有代码结构兼容，不需要推倒重来
- 后续继续扩展外观和控件成本低

缺点：

- 比局部补丁需要更多整理工作

### 方案 C：整体重做播放器外壳

彻底重写 `AppShell`、页面骨架与底部 Dock。

优点：

- 自由度最高

缺点：

- 超出本轮优化范围
- 风险和回归面明显增加

## 推荐方案

采用方案 B：结构化升级，保留现有产品壳。

核心理由：

- 当前项目已经具备较好的页面与播放器基础，不需要推翻
- 用户需求集中在“补齐外观能力 + 稳定桌面布局 + 统一交互语言”
- 现有代码已经有 `theme.less`、`AppShell`、`AppTopbar`、`AppSidebar`、`PlayerDock`、`player store` 等切入点，适合在原骨架上增强

- 左侧视觉高度不再与右侧保持一致
- 底部固定 Dock 与内容区缺少更明确的空间契约

### 按钮与图标

当前多个区域仍使用纯文字作为操作按钮，例如：

- [`src/components/dock/PlaybackControls.vue`](F:\github\my-music-player\src\components\dock\PlaybackControls.vue)
- [`src/components/dock/VolumeControl.vue`](F:\github\my-music-player\src\components\dock\VolumeControl.vue)
- [`src/components/music/TrackTable.vue`](F:\github\my-music-player\src\components\music\TrackTable.vue)

这会让部分高频交互看起来更像文字标签，而不是播放器控件。

### 播放器体验

当前底部 Dock 已经接通真实时间、时长和 seek 事件，但仍存在：

- 进度条视觉反馈还不够“播放器化”
- 控制按钮语义与视觉层级不统一
- 音量、模式、播放状态与主题系统还没有统一抽象

## 架构设计

### 1. 外观状态层

新增独立外观 store，职责如下：

- 管理当前模式：`light` / `dark` / `system`
- 管理当前主题预设：例如 `mist`、`ocean`、`peach`、`graphite`
- 管理自定义主色值
- 负责将状态同步到根节点 CSS 变量或数据属性
- 负责本地持久化

建议新增：

- `src/stores/appearance.ts`
- `src/lib/theme/` 目录，用于整理主题预设、颜色推导与 DOM 应用逻辑

#### 模式策略

为减少第一次实现复杂度，模式层采用以下规则：

- 用户可手动切换亮色 / 暗色
- 可额外预留 `system` 作为兼容入口，但 UI 首版不强依赖
- 亮色和暗色分别定义一套玻璃材质基础 token
- 主题色仅覆盖强调色、按钮高亮、滑块进度、活动态边框、辅助光晕，不直接覆盖整套背景

#### 主题色策略

主题色分两层：

1. 预设主题色
2. 自定义主色

预设主题色用于一键切换常用搭配，自定义主色用于用户进一步微调。实现上，自定义主色优先级高于预设色；当用户清除自定义色时，回退到当前预设色。

#### 变量策略

将现有 `theme.less` 从“单套颜色定义”升级为“主题 token 入口”，分成三类变量：

- 中性色变量：背景、文本、边框、玻璃面板、阴影
- 状态变量：hover、active、selected、focus、disabled
- 品牌变量：主色、主色柔光、主色渐变、进度条高亮、强调边框

这样可以保证亮暗模式切换主要影响中性色，主题色切换主要影响强调层，避免玻璃材质被主题色污染过重。

### 2. 顶部栏外观控制区

[`src/components/chrome/AppTopbar.vue`](F:\github\my-music-player\src\components\chrome\AppTopbar.vue) 右侧从当前静态状态区升级为“外观控制区”，包含：

- 亮 / 暗模式切换按钮
- 主题预设切换按钮组
- 自定义主色入口

交互原则：

- 高优先级操作直接可见，不藏进深层菜单
- 采用图标按钮或图标 + 文字按钮
- 保持桌面播放器工具条感，不做复杂设置页

推荐信息组织：

- 模式切换：太阳 / 月亮按钮
- 主题色：一排小色点按钮
- 自定义主色：调色盘按钮 + 原生颜色选择器或小型浮层入口

### 3. 应用壳层布局

桌面端布局改为“固定视窗内分区滚动”。

推荐结构：

- 根容器固定为视窗高度
- 左侧侧栏与右侧主内容区共享同一内容高度基线
- 右侧主内容区内部再划分为：
  - 顶部栏固定区
  - 主内容滚动区
- 底部 Dock 固定悬浮，但由外层布局预留安全区

#### 布局行为约束

- 桌面端不再依赖整个 `body` 滚动承载内容
- 右侧页面超出时，只在右侧滚动容器中滚动
- 左侧导航区高度固定，内容在可用高度内分布，避免因右侧增长而错位
- 页面底部内容不会被悬浮 Dock 遮挡

#### 左侧侧栏策略

[`src/components/chrome/AppSidebar.vue`](F:\github\my-music-player\src\components\chrome\AppSidebar.vue) 保持固定高度玻璃侧栏，内部采用三段式：

- 顶部品牌区
- 中部主导航区
- 底部辅助区或留白区

主导航在桌面端居中分布，使左侧视觉重心更稳定。

### 4. 图标与按钮系统

新增统一按钮层，覆盖以下场景：

- 导航按钮
- 顶部外观控制按钮
- 播放器传输控制按钮
- 音量和静音按钮
- 喜欢按钮
- 列表行内播放按钮
- 排序按钮

#### 图标策略

统一接入 `Iconify`，并选用一套线性、冷静、与玻璃风匹配的图标语言。

推荐原则：

- 导航项：图标 + 文本
- 高频按钮：图标为主，必要时附带文字
- 空间紧凑场景：图标按钮 + 明确 `aria-label`

#### 按钮语义策略

所有交互位必须使用语义正确的按钮组件或等价交互元素：

- 点击触发动作的地方使用 `button`
- 仅用于展示信息的地方不伪装成按钮
- 列表行若支持整行播放，也要保留单独行内操作按钮，避免操作不明确

#### 样式规则

统一抽象出以下视觉级别：

- `icon-button`：纯图标操作
- `pill-button`：图标 + 文字的圆角按钮
- `primary-control`：播放器主按钮，如播放 / 暂停
- `segmented-button` 或 `filter-button`：排序和模式切换

这层样式应放在全局样式或公共组件中，而不是分散定义在各页面内。

### 5. 底部播放器升级

播放器保持现有悬浮玻璃 Dock 大体结构，但升级交互和视觉统一性。

#### 进度条

[`src/components/dock/PlaybackProgress.vue`](F:\github\my-music-player\src\components\dock\PlaybackProgress.vue) 继续使用真实的 `currentTime`、`duration` 和 `seek` 链路，但要加强：

- 主题色联动的填充轨道
- 更清晰的 thumb 悬停与拖动态
- 禁用态 / 无时长态的明确反馈
- 暗色模式下的对比度调整

如果现有音频事件刷新在视觉上偏跳，可以增加轻量视觉同步优化，但不能脱离真实播放时间。

#### 控制按钮

[`src/components/dock/PlaybackControls.vue`](F:\github\my-music-player\src\components\dock\PlaybackControls.vue) 与 [`src/components/dock/VolumeControl.vue`](F:\github\my-music-player\src\components\dock\VolumeControl.vue) 改为：

- 上一首 / 播放暂停 / 下一首：图标化
- 模式切换：图标 + 简短文字或提示
- 静音：图标按钮
- 音量：图标 + 真实滑块

主播放按钮保持视觉中心，但不能脱离全局按钮语言。

#### 元信息区

歌曲标题、歌手、专辑和状态文本继续保留，用于平衡“全图标化”后的可读性。也就是说，本次不追求无字 UI，而是追求“按钮不再被文字替代”。

## 数据流设计

### 外观流

1. 用户在顶部栏切换模式、主题预设或自定义主色
2. `appearance store` 更新状态
3. 外观应用逻辑将状态映射到根节点属性和 CSS 变量
4. 全站组件通过变量自动更新
5. 状态同步到本地存储

### 播放器流

1. 用户通过页面、侧栏或 Dock 操作播放器
2. `player store` 更新真实播放状态
3. Dock 进度、页面活动态、喜欢状态同步更新
4. 外观系统只负责视觉变量，不介入播放逻辑

## 错误处理

### 外观系统

- 无效的自定义颜色值回退到当前预设主题色
- 本地存储读取失败时回退默认亮色主题
- 不支持颜色输入的环境下，自定义入口可降级为仅预设色

### 图标系统

- 若个别图标未加载成功，按钮保留可点击区域和辅助文案
- 不允许因为图标未渲染导致按钮消失

### 布局系统

- 在窄窗口下允许降级为纵向布局
- 桌面断点优先保证结构稳定，而不是强求所有元素同尺寸

### 播放器

- `duration = 0` 时进度条进入禁用态
- 拖动进度条只改变当前位置，不改变播放 / 暂停状态
- 播放失败提示保持轻量，不扩展为复杂错误面板

## 风险与缓解

### 风险：主题变量扩散，导致组件内硬编码越来越多

缓解：

- 优先建立清晰的 token 层
- 禁止在业务页面内直接写死亮暗模式颜色

### 风险：图标替换后测试选择器或文案断言失效

缓解：

- 保留稳定的 `data-testid` 与 `aria-label`
- 测试优先断言行为和语义，而不是肉眼文案

### 风险：固定高度布局在小窗下变拥挤

缓解：

- 为桌面和窄宽度分别定义布局策略
- 仅在桌面断点强制左右同高

### 风险：自定义主色与玻璃背景耦合过深，导致界面发脏

缓解：

- 将自定义色限制在强调层，不直接染色大面积玻璃主体
- 对深色模式单独校正主色透明度与混色强度
